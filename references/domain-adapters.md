# Domain Adapters — 코딩·문서·콘텐츠별 적용 가이드

러비스킬의 3대 도메인에 대한 분해·평가·상위설계 어댑터

---

## 도메인 판별

| 신호 | 도메인 | 예시 |
|------|--------|------|
| 코드 파일, 함수, 클래스, API, DB 스키마 | **코딩** | .py, .ts, .sql, 아키텍처 다이어그램 |
| 보고서, 기획안, 매뉴얼, 이메일, 정책문서 | **문서** | .md, .docx, .pdf, 위키 |
| 카피, UX 라이팅, 영상 스크립트, 광고, SNS | **콘텐츠** | 랜딩페이지, 유튜브 대본, 뉴스레터 |
| 복합 | **하이브리드** | README(코드+문서), PRD(문서+콘텐츠) |

하이브리드 → 주 도메인 판별 후, 부 도메인 기준도 병행 적용

---

## 1. 코딩 어댑터

### 분해 단위 (러셀 원자론 적용)

| 분해 단위 | 대응 | 원자적 질문 |
|-----------|------|-----------|
| 함수/메서드 | 원자 명제 | 이 함수는 하나의 일만 하는가? |
| 변수/상수 | 기호(symbol) | 이 이름이 가리키는 것이 명확한가? |
| 클래스/모듈 | 유형(type) | 추상 수준이 일관적인가? |
| 의존성 | 관계(relation) | 순환 참조가 없는가? |
| 조건문 | 명제 논리 | 모든 경우를 빠짐없이 다루는가? |
| 설정/환경변수 | 공리(axiom) | 기본 가정이 명시되어 있는가? |

### 평가 기준

| 6축 | 코딩에서의 의미 | 검증 방법 |
|-----|---------------|---------|
| 목적성 | 이 코드가 왜 존재하는가 | 주석/커밋 메시지에 "왜"가 있는가 |
| 방향성 | 데이터 흐름이 입력→처리→출력으로 수렴하는가 | 콜 그래프 추적 |
| 정합성 | 타입 에러, 논리 모순, 데드 코드 없는가 | 정적 분석, 타입 체크 |
| 최소성 | 중복 코드, 불필요한 추상화 없는가 | 코드 중복 탐지, Russell's Razor |
| 확장성 | 새 기능 추가 시 기존 코드 수정 최소화되는가 | OCP(개방-폐쇄 원칙) 체크 |
| 사용적합성 | 팀이 실제로 유지보수할 수 있는가 | 코드 리뷰 통과 여부, 온보딩 난이도 |

### 상위 설계 패턴

| 현재 상태 | 상위 설계 방향 | 기법 |
|-----------|--------------|------|
| 거대 함수 | 원자 함수 조합 | Extract Method → Compose |
| 중첩 조건문 | 선언적 규칙 | Guard Clause, Strategy Pattern |
| 중복 로직 | 공유 추상화 | DRY → 그러나 잘못된 추상화 경계 주의 |
| 암묵적 의존성 | 명시적 주입 | DI, Configuration as Code |
| God Object | 책임 분리 | SRP, 도메인 모듈화 |

## 2. 문서 어댑터

### 분해 단위 (러셀 원자론 적용)

| 분해 단위 | 대응 | 원자적 질문 |
|-----------|------|-----------|
| 문장 | 원자 명제 | 이 문장은 하나의 주장만 하는가? |
| 문단 | 복합 명제 | 이 문단의 핵심 명제는 무엇인가? |
| 섹션 | 유형(type) | 같은 추상 수준의 내용인가? |
| 용어 | 기호(symbol) | 문서 전체에서 일관된 의미인가? |
| 논증 구조 | 추론 규칙 | 전제→결론이 논리적인가? |
| 전제/가정 | 공리(axiom) | 명시되어 있는가? |

### 평가 기준

| 6축 | 문서에서의 의미 | 검증 방법 |
|-----|---------------|---------|
| 목적성 | 이 문서의 독자와 목표가 명확한가 | 첫 문단에 "이 문서는 X를 위해" |
| 방향성 | 결론으로 수렴하는가 | 목차→결론 경로 추적 |
| 정합성 | 앞뒤 모순, 용어 불일치 없는가 | 교차 참조 체크 |
| 최소성 | 같은 말 반복, 불필요한 섹션 없는가 | 각 문단 삭제 테스트(빼도 되는가?) |
| 확장성 | 내용 추가 시 구조가 유지되는가 | 새 섹션 삽입 시뮬레이션 |
| 사용적합성 | 독자가 실제로 읽고 행동할 수 있는가 | 비트겐슈타인 언어게임 체크 |

### 상위 설계 패턴

| 현재 상태 | 상위 설계 방향 | 기법 |
|-----------|--------------|------|
| 나열식 | 수렴 구조 (모래시계) | 핵심 명제 → 근거 → 결론 |
| 용어 혼재 | 용어 통일 | 용어집(glossary) 선행, 언어게임 일치 |
| 전제 암묵적 | 가정 명시 | "이 문서는 X를 가정한다" 선언 |
| 과잉 서술 | 압축 | Russell's Razor — 논리적 구성으로 대체 |
| 독자 미고려 | 맥락 적합 | 삶의 형식 검증 — 독자 환경에서 테스트 |

## 3. 콘텐츠 어댑터

### 분해 단위 (러셀 원자론 적용)

| 분해 단위 | 대응 | 원자적 질문 |
|-----------|------|-----------|
| 카피/문장 | 원자 명제 | 이 문장의 메시지는 무엇인가? |
| 훅/오프닝 | 전제 | 관심을 잡는 전제가 유효한가? |
| 전개 | 추론 규칙 | 훅에서 결론으로의 논리가 성립하는가? |
| CTA/결론 | 정리(theorem) | 행동 유도가 명확한가? |
| 톤/보이스 | 언어게임 | 대상 독자의 "게임"에 맞는가? |
| 비주얼 | 그림이론 | 이미지가 메시지와 동형인가? |

### 평가 기준

| 6축 | 콘텐츠에서의 의미 | 검증 방법 |
|-----|-----------------|---------|
| 목적성 | 이 콘텐츠가 유도하는 행동이 명확한가 | CTA 식별 |
| 방향성 | 훅→전개→CTA 흐름이 자연스러운가 | 읽기 흐름 추적 |
| 정합성 | 톤/메시지/비주얼이 일관적인가 | 크로스 체크 |
| 최소성 | 핵심 메시지가 한 문장으로 압축되는가 | 엘리베이터 테스트 |
| 확장성 | 다른 채널/포맷으로 변환 가능한가 | 멀티채널 적용 시뮬레이션 |
| 사용적합성 | 타겟 오디언스가 실제로 반응하는가 | 비트겐슈타인 삶의 형식 |

### 상위 설계 패턴

| 현재 상태 | 상위 설계 방향 | 기법 |
|-----------|--------------|------|
| 메시지 산만 | 핵심 1메시지 | "한 문장으로 말하면?" 테스트 |
| 훅 약함 | 긴장 구조 | 대비·질문·도발로 시작 |
| CTA 불분명 | 행동 명시 | 동사+구체적 대상 |
| 톤 불일치 | 언어게임 재설정 | 타겟의 실제 언어 조사 |
| 길어서 안 읽힘 | 압축 + 구조화 | Russell's Razor + 시각적 위계 |

## 하이브리드 처리

복합 도메인(예: README = 코딩+문서, PRD = 문서+콘텐츠) 처리:

1. **주 도메인 판별**: 핵심 기능이 속하는 도메인
2. **부 도메인 식별**: 보조 기능이 속하는 도메인
3. **주 도메인 기준으로 분해·평가** → 부 도메인 기준으로 보완 검증
4. **충돌 시**: 사용적합성(비트겐슈타인 축)이 우선 — "실제로 쓰이는 맥락"이 최종 판정

## 디테일 보존 원칙

**러비스킬의 핵심 약속: 디테일을 놓치지 않는다**

| 단계 | 보존 방법 |
|------|---------|
| 분해 | 원본의 모든 요소를 매핑 테이블에 기록 → 누락 체크 |
| 평가 | 매핑 테이블의 각 요소별로 6축 판정 → 건너뛰기 금지 |
| 상위설계 | 제거하는 요소에 대해 "왜 제거하는가" 명시 → 무단 삭제 금지 |

**실행 게이트**: 분해 단계에서 원본 요소 목록을 먼저 출력하고, 그 목록이 완전한지 확인한 후에만 평가·상위설계 진행. 목록이 불완전하면 STOP.
